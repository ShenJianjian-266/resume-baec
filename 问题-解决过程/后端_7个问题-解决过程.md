# 后端项目面试讲述：7条「问题-解决过程」

项目：美团商业分析精英赛宣传平台（后端：`ba-main-platform-service/`；前端：`ba-main-platform-fe/`）

---

## 1) 问题：前后端联调时，返回结构不统一导致前端难以稳定处理成功/失败

**解决过程**
- 先在接口文档里统一约定通用响应结构（`code/message/data/timestamp`），并固化错误码含义。
- 后端实现统一响应包装类 `ApiResponse`，提供 `success / paramError / notFound / serviceError / dbError` 等工厂方法（`ba-main-platform-service/src/main/java/com/sankuai/ba/bamainplatformservice/common/ApiResponse.java`）。
- 控制器层全部以 `ApiResponse<T>` 对外返回，确保任何接口都能被前端以相同方式解析（`.../controller/*Controller.java`）。
- 前端请求侧统一按 `payload.code === 0` 判断成功并抛错（示例：`ba-main-platform-fe/src/sections/home/homeCardsApi.js`、`ba-main-platform-fe/src/sections/history/historyApi.js`）。

**你可以这样讲（1句话）**
- "我先把接口的'成功/失败'规范定下来，然后用统一 `ApiResponse` 落地，前端只需判断 `code` 就能稳定联调。"

## 2) 问题：同一套代码要跑 dev/test/staging/prod，多环境配置容易混用且上线风险高

**解决过程**
- 通过 `MdpContextUtils.getHostEnvStr()` 在启动时自动识别宿主环境，并加载对应 profile（`ba-main-platform-service/src/main/java/com/sankuai/ba/bamainplatformservice/ApplicationLoader.java`）。
- 把环境差异配置下沉到 `src/main/profiles/<env>/`，分别维护 `application.properties / zebra.properties / s3plus.properties / log4j2.xml / sso.properties`，避免"一个配置文件改来改去"（目录：`ba-main-platform-service/src/main/profiles/`）。
- 健康检查、端口等基础能力在各环境的 `application.properties` 明确声明，保证发布系统判活一致（例如 `management.endpoints.web.base-path=/monitor`、`health=/alive`）。

**你可以这样讲（1句话）**
- "我把环境识别和配置加载自动化：启动时按宿主环境选 profile，配置按环境分目录，降低线上误配概率。"

---

## 3) 问题：Zebra 管数据源时，Spring Boot 默认数据源自动配置会冲突，导致启动/事务问题

**解决过程**
- 启动类显式排除 `DataSourceAutoConfiguration`，把数据源完全交给 Zebra（`@SpringBootApplication(exclude={DataSourceAutoConfiguration.class})`）。
- 启用事务管理 `@EnableTransactionManagement`，交由 Zebra 事务管理器处理（`ApplicationLoader.java`）。
- 通过 `@MapperScan` + `mdp.zebra.basePackage` 统一扫描 Mapper，避免"有的 Mapper 被扫到、有的没被扫到"的隐性问题（`ApplicationLoader.java`、`src/main/profiles/*/zebra.properties`）。

**你可以这样讲（1句话）**
- "我明确了数据源边界：Spring 默认数据源不接管，Zebra 统一管理连接池和事务，启动更可控。"

---

## 4) 问题：MyBatis 配置要求 mapperLocations，但项目又希望用注解 SQL 减少 XML 维护成本

**解决过程**
- Mapper 接口使用 `@Select` 直接写 SQL（`ba-main-platform-service/src/main/java/com/sankuai/ba/bamainplatformservice/mapper/*.java`）。
- Zebra 配置里仍保持 `mdp.zebra.mapperLocations=classpath:mappers/*.xml`（`src/main/profiles/*/zebra.properties`）。
- 为避免启动时因找不到 XML 报错，加入 `mappers/placeholder.xml` 作为占位（`ba-main-platform-service/src/main/resources/mappers/placeholder.xml`），同时把 SQL 的"单一事实来源"放在 Mapper 注解里。

**你可以这样讲（1句话）**
- "框架要 XML，但我不想维护一堆 XML：用注解 SQL + 一个占位 XML，既满足框架约束又降低维护成本。"

---

## 5) 问题：列表页需要快，但赛事/资讯详情包含富文本大字段，直接放主表会拖慢列表查询

**解决过程**
- 数据库层按"主表/内容表"拆分：`competition` 存核心字段，`competition_content` 存富文本（设计说明见 `doc-bmp/plan/database/数据库设计文档_v1.0.md`，初始化脚本见 `ba-main-platform-service/src/main/resources/db/init_competition.sql`）。
- 接口层按场景取数：
  - 历届列表仅查主表轻字段并分页（`CompetitionMapper.selectHistoryCompetitions` + `CompetitionService.getHistoryCompetitions`）。
  - 详情/当届页再按需补查内容表（`CompetitionMapper.selectContentByCompetitionId` + `CompetitionService.getCurrentCompetition/getHistoryCompetitionDetail`）。
- 所有查询统一带 `is_deleted = 0`，用软删除保证内容可回收、数据可追溯（SQL 见各 `@Select`）。

**你可以这样讲（1句话）**
- "我用'主表轻字段 + 内容表大字段'拆表，让列表接口只查必要字段，详情再补内容，性能和结构都更清晰。"

## 6) 问题：历届获奖结果展示是多表聚合，还要按奖项等级分组输出给前端渲染

**解决过程**
- 先查获奖结果列表（`AwardMapper.selectAwardResultsByCompetitionId`）。
- 再把奖项配置转成 Map，避免后续反复查奖项元数据（`CompetitionService.getHistoryCompetitionAwards` 内 `prizeConfigMap`）。
- 用 `LinkedHashMap` 按 `prizeLevel` 聚合成 `AwardGroupDTO`，并在每组内拼装团队与成员信息，最终按 `prizeLevel` 排序输出（`CompetitionService.getHistoryCompetitionAwards`）。
- 前端可以直接按"奖项分组 → 团队列表 → 成员列表"渲染，不需要在浏览器里做复杂 join（接口：`GET /baec/api/v1/competition/history/{id}/awards`）。

**你可以这样讲（1句话）**
- "这个接口本质是'多表 join + 业务分组'：我在服务端把数据聚合成前端友好的分组结构，前端拿到就能直接渲染。"

## 7) 问题：上线需要自动判活与基础可观测，否则发布系统无法可靠确认服务可用

**解决过程**
- 在各环境 `application.properties` 里固定健康检查入口：`/monitor/alive`（`management.endpoints.web.base-path=/monitor` + `health=/alive`）。
- 发布侧提供 `deploy/check.sh` 按 `TEST_URL` 轮询判活，默认指向 `127.0.0.1:8080/monitor/alive`，失败即阻断发布（`ba-main-platform-service/deploy/check.sh`）。
- `plusboot.yaml` 固化 JDK/Maven 版本与 GC 日志、G1 参数，减少"同代码不同容器表现不一致"的排查成本（`ba-main-platform-service/plusboot.yaml`）。

**你可以这样讲（1句话）**
- "我把'可发布'当成需求的一部分：健康检查路径固定 + 发布脚本判活 + 运行参数固化，上线更稳。"
