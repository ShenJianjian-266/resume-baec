##### 解决"加载首页主视觉区画面慢"的问题，速度提升60%
问题：首页主视觉区画面在PC端和移动端下分别是视频和图片。它们是通过S3Plus对象存储服务来存储的。在使用Chrome开发者工具硬刷新首页时发现，加载首页主视觉区画面很慢且有明显卡顿，从“路由进入 -> 首页主视觉完整可见”的耗时为“PC端下93ms左右 移动端下66ms左右”。
解决过程：
    我从学城文档检索发现一个mini课堂项目的视频资源也是通过S3Plus对象存储服务来存储的，同时线上的S3Plus对象存储服务可以使用CND加速，而他就使用到了CND加速来优化加载速度。我通过查阅资料了解到使用CND加速的好处，包括用户体验好、成本更低、更加安全、异地容灾。
        （1）用户体验好：经过调度和缓存实现源站加速，内容服务更靠近用户，使用户在打开图片、视频、直播等多媒体服务时，体验到秒开的流畅感；
        （2）成本更低：源站公网带宽成本是CDN带宽成本的8倍，同时CDN边缘内容缓存命中率一般在95%+，可以极大节省源站带宽成本；
        （3）更加安全：服务更多的沉淀到了CDN边缘节点，相当于为源站增加了一层广泛分布的保护伞，此外CDN厂商在边缘侧也有相应的防护服务，可以抵御大部分互联网攻击
        （4）异地容灾：头部CDN厂商在国内的边缘节点分布都有1000+，单地域的网络/机房/服务器故障，可以结合DNS切换快速规避对用户体验的影响;
    于是我申请了线上的S3Plus对象存储服务的bucket，使用了基于CND加速的S3Plus对象存储服务来存储首页主视觉区的视频和图片，加载首页主视觉区画面的耗时并未明显减少。使用Chrome DevTools（Network 勾选 Disable cache）硬刷新首页，依旧能感受到加载首页主视觉区画面会很慢。
    为了解决这个问题，我重新分析了一下首页主视觉区的代码，发现造成这个问题的原因有3点：
        （1）移动端会先误加载 PC 首屏。`HomeView.vue` 里 `isMobile` 初始为 `false`，到 `mounted()` 才判断宽度；移动端首帧先渲染 PC 分支并触发 **MP4 请求**，再切到移动端重新加载一次资源。这导致首屏明显变慢、还浪费流量。
        （2）PC端 视频没有 `poster`。视频首帧没到之前只能看到底色（`background: #0b0f14`），用户体感是“在等”。
        （3）关键媒体请求发起太晚。媒体 URL 写在组件里，只有等 JS 下载/执行 + Vue 渲染后才发起请求；`index.html` 没有 `preconnect/preload`。
        （4）媒体文件太大，比如卡片背景图大小最大达到了7.13MB。
    找到问题原因后，我基于这些原因进行了优化：
        （1）移动端不先加载 PC，同步引入双端组件，首次渲染前就判定端。
```js
import HomeViewPc from './HomeView.pc.vue'
import HomeViewMobile from './HomeView.mobile.vue'

const isMobileNow = () => (typeof window !== 'undefined' ? window.innerWidth <= 768 : false)

export default {
  components: { HomeViewPc, HomeViewMobile },
  data() {
    return { isMobile: isMobileNow() }
  },
  // mounted 仍保留 resize 监听即可
}
```     
        (2)PC端视频加 `poster`，先出静图，再播视频。这样改动后，就算网络再慢，首屏也能先看到 poster，不再“黑底等视频首帧”。
```js
在 `ba-main-platform-fe/src/sections/home/HeroSection.pc.vue`:1. `<video>` 增加 `:poster="posterSrc"`  2. `preload` 从 `auto` 改成 `metadata`（避免 MP4 抢带宽影响静图到达）
<video
  class="hero-video"
  :src="videoSrc"
  :poster="posterSrc"
  autoplay
  muted
  loop
  playsinline
  preload="metadata"

。。。

  return {
    videoSrc: HERO_PC_VIDEO_URL,
    posterSrc: HERO_PC_POSTER_URL,
  }
}
```
        (3)在 index.html 提前 preconnect + preload（让静图更早开始下载）。
```js

<link rel="dns-prefetch" href="https://s3plus.meituan.net" />

<!-- 2) 首屏关键静图：PC 预加载 poster，移动端预加载背景图 -->
<link
  rel="preload"
  as="image"
  href="https://s3plus.meituan.net/ba-main-platform-pub/home-hero-poster.webp"
  media="(min-width: 769px)"
/>
<link
  rel="preload"
  as="image"
  href="https://s3plus.meituan.net/ba-main-platform-pub/home-hero-app.png"
  media="(max-width: 768px)"
/>
```
        (4)缩小媒体文件大小。
效果：经过这些优化后，加载首页主视觉区画面的速度比优化前快了很多。从“路由进入 -> 首页主视觉完整可见”的耗时由原来的“PC端下93ms左右 移动端下66ms左右”减少至“PC端下37ms左右 移动端下18ms左右”。

##### 解决前端发布部署后无法访问问题
问题：talos是一个可以自定义发布流水线模版发布部署webstatic类型项目的平台，我在talos上成功发布部署了前端项目后，在浏览器中输入地址访问网站时出现了页面无法访问的情况。
解决过程：
    我首先查看页面的详细信息，看到ResourceNotFound，就去WebStatic 静态网站托管服务平台上查看了资源浏览器，发现资源是存在的，说明成功发布了资源。查阅了WebStatic文档后，发现自己遗漏了“路由重写&重定向”。就按照要求配置了正确的路由重写规则。在浏览器中输入地址访问网站，ResourceNotFound消失了，但是页面是空白。
    我先查看了WebStatic常见问题文档，没有找到我这个问题，在学城中检索所有文档也没有相关的文档。于是我就向平台值班人员提了这个问题，他所给出的解决方案都是已经实现了的，并没有实际解决掉问题。
    后来我就看了组里另一个项目的前端代码，发现其路由中history为createWebHashHistory()，这和我负责的项目的前端代码不同，我的是createWebHistory()。我试着将其改为createWebHashHistory()重新发布部署后，就能正常访问了。
    至于为什么。我询问了处理我这个问题的人员后，也没有得到具体答案。
    【为什么？下面是codex答案，并未解决疑问】
```
为什么 history 会空白、hash 为什么能好。

原因本质上不在 Chrome，而在：**history 模式对部署环境有额外要求**。常见命中点有两类（命中任意一种都可能出现“白屏/空白”）。

1) 服务器/网关没做 SPA 回退（history-fallback / rewrite）
- `createWebHistory()` 的 URL 是 `/history`、`/news/123` 这种“真实路径”。
- 用户直接打开/刷新这些地址时，浏览器会向服务器请求 `GET /history`。
- 若服务器没有把“所有非静态资源路径”统一重写回 `index.html`（而是按传统后端路由去找 `/history`），就会 404/返回错误页，前端应用无法启动，表现为“空白”。

2) 打包后的静态资源路径是相对路径，导致深路径下 JS/CSS 404
- 若构建产物 `index.html` 里引用资源是 `./assets/xxx.js` 或 `assets/xxx.js` 这类相对路径：
  - history 模式打开 `/history` 时，资源会被解析成 `/history/assets/xxx.js`，通常 404，页面就空白。
  - hash 模式打开 `/#/history` 时，请求路径仍然是 `/`（`#` 后面的内容不会发给服务器），资源解析为 `/assets/xxx.js`，因此“换成 hash 就好了”。

为什么换成 hash 就能好
- `createWebHashHistory()` 把路由放到 `#` 后面（如 `/#/history`），`#` 片段不会参与 HTTP 请求路径。
- 服务器永远只需要返回 `/` 对应的 `index.html`，不需要额外配置重写规则，也天然规避“深路径下相对资源 404”。

如何快速确认到底是哪一种
- 打开 Chrome DevTools → Network/Console，重点看：
  - `/history` 直接 404（或返回的不是 `index.html`）→ 更像第 1 类（缺少 SPA 回退重写）。
  - `/history/assets/...`、`/news/assets/...` 404，或 Console 报 `Failed to load module script` / `Unexpected token '<'`（JS 实际拿到的是 HTML）→ 更像第 2 类（相对资源路径在深路径下失效）。

如果想继续使用“更干净的 URL”（history 模式）
- 部署侧：加 SPA 回退重写到 `index.html`（所有前端路由都回退到入口页）。
- 构建侧：保证 `base/PUBLIC_PATH` 与部署路径一致，避免相对资源在深路径下丢失。
```
##### 解决埋点记录过多导致系统性能降低问题
问题：每一次打开、关闭、点击等都会记录埋点到数据库中，随着时间的推移，表中的记录会越来越多达到一个很大的数量。我们统计页面浏览量、导航项点击量等指标需要查询数据库，若数据库中记录量很大，则查询数据库就会很慢，从而导致系统的响应速度很慢，影响用户体验。同时，统计时并不需要包含时间久远的记录，这造成了数据库中的数据冗余问题。
解决过程：为了解决这个问题，我决定增加一个定时删除记录的功能。首先，我查阅了RDS文档，发现RDS并不支持设置定时删除记录的功能。所以我只能在后端代码层面实现这个功能。过程包括开启Spring定时任务能力、在Mapper增加“按日期批量删除”方法、新增定时任务类（循环批量删）、增加配置。
    （1）开启 Spring 定时任务能力。在ApplicationLoader增加注解@EnableScheduling。
    （2）在Mapper增加“按日期批量删除”方法。对应的sql语句的where子句为`where event_date<目标date and limit 目标limit`。这里使用limit是避免一次删除过多导致大事务、长锁、主从延迟抖动，通过“多批次循环”删完。
    （3）新增定时任务类（循环批量删）。使用Mapper新增的方法来循环批量删除过期埋点记录。
    （4）增加配置。在各个环境的配置文件中新增配置字段用来控制是否开启定时删除、记录保持时间、每批删除的记录数量、最大删除的批次数量、执行定时任务的时机。这些字段会被新增的定时任务类通过@Value注解使用。
收尾：以上就是关于解决埋点记录过多导致系统性能降低的问题的情况。

##### 添加缓存服务解决数据请求响应慢的问题，
问题：原来所有的查询数据请求都会直接查询数据库，这样做会导致请求响应速度慢。
解决过程：我在学城中查阅资料，发现接入美团内部的squirrel服务就可以使用redis缓存服务。于是我参考squirrel文档接入了squirrel服务，主要步骤包括申请资源、pom添加依赖、创建 SquirrelConfig 配置类、配置各环境 application.properties、在合适位置使用缓存。
    （1）申请资源。在 Squirrel 管理平台申请 cluster 和 category。每一类缓存数据对应一个 category，类比数据库的一张表。
    （2）pom添加依赖。在pom.xml中添加squirrel-client依赖。
    （3）创建 SquirrelConfig 配置类。包含获取bean的方法，主要用于获取CacheHelper。CacheHelper类中包含一个带互斥锁的获取缓存的方法，它能预防缓存击穿问题。核心思路是「让同一 key 在缓存失效时，只允许 **1 个线程** 去查 DB，其余线程排队等待，拿到锁后再查一次缓存（此时已由第一个线程填充好），若命中则返回，否则更新缓存。」
    （4）配置各环境 application.properties。添加squirrel配置，包括clusterName和readTimeout。
    （5）在合适位置使用缓存。就是在Service中注入CacheHelper并使用其获取缓存的方法。
收尾：以上就是关于解决添加缓存服务解决数据请求响应慢的问题以及预防缓存击穿问题的情况。

##### 使用nocode平台开发富文本编辑器方便运营同学编辑资讯内容、提高生产效率
问题：像赛事介绍、比赛规则、比赛资讯内容这些在数据库中都是富文本类型的字段，若内容带有图片或表格等复杂结构，则原始代码数据就会包含有各种标签和样式，若运营同学需要新增/编辑这些内容，则要先学习理解各种标签样式代码，从而降低了生产效率。
解决过程：nocode平台是一款无需编程背景和经验，以自然语言或图片对话形式，即可轻松创建应用的工具。为了快速完成开发富文本编辑器工具，我决定使用nocode平台开发。经历11个版本迭代，开发出一款功能强大、界面友好的富文本编辑器。比如支持表格调整行高、列宽、设置表头行、设置表头列；支持图片调整对齐方式、宽度占比、圆角大小。同时这款编辑器的工具栏、表格右键菜单项等与学城文档编辑器保持一致，大大方便了运营同学上手，运营同学只需像写文档一样编辑富文本内容，然后切换到代码查看界面把对应的代码更新到数据库中即可。
收尾：以上就是关于使用nocode平台开发富文本编辑器方便运营同学编辑资讯内容、提高生产效率的情况。
